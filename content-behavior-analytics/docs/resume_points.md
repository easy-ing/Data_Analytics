# 자소서 연결 핵심 성과 포인트

## 1) 지원동기 문항 연결
- 콘텐츠 소비 행동이 서비스 핵심 지표(CTR, 체류시간, 참여율)에 직결된다는 문제의식에서 프로젝트를 시작.
- 단순 대시보드 조회가 아니라, 로그 생성부터 모델링/적재/분석까지 직접 구현해 "데이터로 의사결정 가능한 구조"를 만드는 데 집중.
- 네이버처럼 대규모 트래픽 환경에서 중요한 이벤트 표준화와 KPI 정의의 중요성을 체감.

## 2) SQL 역량 문항 연결
- 이벤트 로그 기반 KPI SQL 10종을 직접 작성.
- `CTE`, `FILTER`, `CASE WHEN`, `COUNT(DISTINCT)`, `NULLIF`, `ON CONFLICT` 등 실무형 문법 사용.
- 퍼널 전환률, 코호트 리텐션, 채널별 CTR, 가중치 랭킹 등 단순 집계를 넘어 비즈니스 질문에 대응하는 분석 쿼리 설계.
- 사전 집계 마트(`mart_content_daily`)를 활용해 반복 조회 성능과 분석 일관성을 개선.

## 3) 데이터 모델링 경험 문항 연결
- `dim_users`, `dim_content`, `fact_user_events`, `mart_content_daily`의 4계층 구조 설계.
- 원천 이벤트(fact)와 분석용 마트(mart)를 분리해 확장성과 유지보수성을 확보.
- 날짜 생성 컬럼(`event_date`)과 이벤트/날짜 인덱스를 추가해 시계열 분석 성능을 고려.
- 콘텐츠/사용자 차원을 분리해 새로운 속성 추가 시 스키마 변경 영향을 최소화.

## 4) 정량 성과로 쓰기 좋은 문장 템플릿
- "30일치 사용자 이벤트 약 N만 건을 합성 생성하고, ETL 파이프라인으로 PostgreSQL 적재를 자동화했습니다."
- "핵심 KPI SQL 10종을 설계해 DAU, CTR, 리텐션, 참여율을 일관된 정의로 재현 가능하게 만들었습니다."
- "차원/팩트/마트 분리 모델링을 적용해 반복 분석 쿼리 시간을 단축하고, 지표 해석의 일관성을 높였습니다."

## 5) 면접에서 강조할 기술적 의사결정
- 왜 raw 이벤트를 그대로 보지 않고 일별 마트를 병행했는지
- CTR/Engagement 정의를 어떻게 고정해 팀 내 지표 충돌을 줄였는지
- 신규 지표 요청 시 어떤 테이블을 수정하면 되는지(확장 경로)
